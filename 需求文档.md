文档 OCR Web 产品 MVP 需求文档

1. 项目概述

1.1 项目名称

DocFlow OCR

1.2 项目目标

构建一个面向非技术用户的 Web 产品，支持上传 PDF 或图片，自动完成 OCR 识别，并提供在线校对、结构化提取与导出能力。

1.3 产品定位

这不是一个“模型演示页”，而是一个 document-to-review workflow（文档处理工作流）产品。
用户目标不是“调用 OCR 模型”，而是“把文档变成可编辑、可导出、可入库的内容”。

1.4 底层能力前提

系统默认以 DeepSeek-OCR-2 为 OCR 引擎，优先使用服务化方式接入；本期不要求用户接触模型参数、Prompt、GPU 环境等技术细节。官方仓库已给出图片与 PDF 推理脚本，以及 vLLM / Transformers 两种接入方式。 ￼

⸻

2. 产品目标与非目标

2.1 本期目标

本期 MVP 需要完成以下核心闭环：

1. 用户上传 PDF / 图片
2. 系统异步执行 OCR
3. 返回可编辑结果
4. 用户在线校对
5. 导出 Markdown / TXT / JSON / Excel

2.2 非目标

本期明确不做：

1. 聊天式助手
2. 多模型路由
3. 企业级审批流
4. 多租户复杂权限体系
5. 复杂模板市场
6. 训练/微调模型
7. 公共 API 开放平台

⸻

3. 目标用户

3.1 主要用户

1. 行政/运营人员：处理扫描件、发票、合同、表单
2. 研究/内容人员：把论文、手册、截图转为 Markdown
3. 小团队内部知识整理人员：把历史 PDF 转成可检索文本

3.2 用户特征

1. 不懂代码
2. 不关心模型实现
3. 只关心上传是否方便、结果是否可改、导出是否可用

⸻

4. 用户场景

场景 A：通用文档转文本

用户上传 PDF，系统输出 Markdown/Text，用户可修正后导出。

场景 B：图片/截图识别

用户上传截图、拍照件，系统提取文字并保留基础结构。

场景 C：结构化字段提取

用户上传发票/合同/表单，系统先 OCR，再按模板提取 JSON 字段，用户确认后导出。

场景 D：批量归档

用户一次上传多个文件，系统异步排队处理，并可逐个查看结果。

6. 功能需求

6.1 上传模块

功能说明

用户可在 Web 页面上传单个或多个文件，选择处理模式后提交任务。

具体要求

1. 支持拖拽上传
2. 支持点击选择文件
3. 支持多文件上传
4. 上传前显示文件名、大小、页数（PDF 可解析页数）
5. 用户可选择处理模式
6. 用户可选择导出目标格式
7. 上传成功后自动创建任务

异常处理

1. 不支持格式时报错
2. 文件过大时报错
3. 上传失败可重试

⸻

6.2 任务管理模块

功能说明

系统采用异步任务机制处理 OCR，用户可查看任务状态。

任务状态

1. uploaded
2. queued
3. processing
4. completed
5. failed
6. canceled

具体要求

1. 用户提交后立即返回任务 ID
2. 任务页显示处理进度
3. PDF 任务需要显示页级进度
4. 失败任务展示失败原因
5. 支持手动重试失败任务

⸻

6.3 OCR 识别模块

功能说明

系统调用 OCR 引擎完成文档识别。

具体要求

1. 支持图片 OCR
2. 支持 PDF 拆页后逐页 OCR
3. 支持输出 Markdown/Text
4. 支持基础版面保留
5. 支持批量任务
6. 支持配置不同 OCR Prompt/模式
7. 支持把页级结果合并为文档级结果

技术约束

底层 OCR 引擎为 DeepSeek-OCR-2；实现层面应优先走服务化调用，而不是把模型逻辑直接塞进 Web 进程。官方仓库已提供图片与 PDF 推理示例。 ￼

⸻

功能说明

3. 支持字段级人工修改
4. 支持字段缺失标记
5. 支持置信度低字段高亮
6. 支持重新抽取

9. 非功能需求

9.1 性能

1. 小图片上传后应能快速进入排队状态
2. PDF 必须异步处理
3. 前端不得因 OCR 推理阻塞页面
4. 结果页打开时应支持懒加载页内容

9.2 可用性

1. 失败任务可重试
2. 页面刷新后任务状态不丢失
3. 编辑结果必须可恢复
4. 大文件处理时应有明确进度反馈

9.3 安全性

1. 文件存储需与业务数据隔离
2. 用户只能访问自己的文件
3. 删除任务后应支持同步删除文件或进入回收站
4. 日志中不得记录敏感全文内容

9.4 可维护性

1. OCR 服务与 Web 服务解耦
2. 所有任务状态变更需可追踪
3. OCR Prompt 和模板配置化
4. 允许未来替换底层 OCR 引擎

10. 技术方案要求

10.1 推荐架构

1. 前端：Next.js
2. 后端 API：FastAPI
3. 数据库：PostgreSQL
4. 队列：Redis + Celery / RQ
5. 文件存储：S3 / MinIO
6. OCR 服务：独立 GPU Worker + DeepSeek-OCR-2
7. PDF 解析：PyMuPDF
8. 图像预处理：Pillow / OpenCV

10.2 架构原则

1. Web 服务不直接加载大模型
2. OCR 推理放在独立 Worker 或独立服务中
3. PDF 先拆页，再页级处理，再合并
4. 前后端通过任务 ID 轮询或 SSE 获取进度
5. 结果存储分为原始 OCR 结果与用户修订结果两份

10.3 模型接入要求

1. 预留 vLLM 服务接入层
2. 预留 transformers 本地调用层
3. 业务代码不得与具体模型实现强耦合
4. OCR Prompt 应配置化
5. 模型服务应可替换为其他 OCR engine

官方仓库说明 DeepSeek-OCR-2 支持 vLLM Inference 与 Transformers Inference，因此系统设计应保留这两种接入抽象，而不是把实现写死。 ￼

13. 任务处理流程

13.1 图片流程

1. 用户上传图片
2. 创建任务
3. OCR Worker 拉取任务
4. 调用 OCR 模型
5. 保存原始结果
6. 返回结果页供用户校对
7. 用户导出

13.2 PDF 流程

1. 用户上传 PDF
2. 创建任务
3. 系统拆页
4. 页级并发 OCR
5. 合并文档结果
6. 生成结构化结果（如有模板）
7. 用户校对
8. 导出

⸻

14. 验收标准

14.1 MVP 必须满足

1. 能上传图片和 PDF
2. 能创建异步 OCR 任务
3. 能看到任务状态和进度
4. 能查看并编辑 OCR 结果
5. 能导出 Markdown / TXT / JSON
6. 失败任务能重试
7. 历史任务可查看

14.2 不通过条件

1. 上传后无法追踪状态
2. PDF 处理无法按页返回
3. 结果不可编辑
4. 导出结果与编辑结果不一致
5. 任务失败后没有错误反馈

16. 开发约束给 Codex

1. 先做 MVP，不要超范围
2. 所有模块按可替换架构设计
3. OCR 引擎必须抽象为 provider interface
4. 先完成单用户闭环，再做多用户优化
5. 所有长耗时任务都必须异步
6. 页面先重功能，后重样式
7. 所有导出都基于“用户修订后的最终结果”
8. 不要把模型代码直接耦合在前端或 Web API 主进程里
